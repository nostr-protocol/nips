NIP-C8
======

Static Websites
---------------

`draft` `optional`

This NIP describes a method fo publishing and serving static websites.

### Site Manifest

The event kind `35128` uses `path` tags to map absolute file paths
to the hex-encoded SHA-256 hash of the corresponding file. All files
are stored on [Blossom](https://github.com/nostr-protocol/nips/blob/master/B7.md).

A `path` tag set to `index.html` is the website `root` path and is mandatory.

The `d` tag SHOULD be the uploaded website folder's name and MAY be a random string.

The event may include metadata on `title`, `description`,
`source` and `server` tags as shown on the following example:

```jsonc
{
  "kind": 35128,
  "tags": [
    ["d", "my-website"],
    ["path", "index.html", "186ea5fd14e88fd1ac49351759e7ab906fa94892002b60bf7f5a428f28ca1c99"],
    ["path", "about.html", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456"],
    ["path", "favicon.ico", "fedcba0987654321fedcba0987654321fedcba0987654321fedcba0987654321"],
    ["path", "assets/main.js", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456"],
    ["title", "My Nostr Site"],
    ["description", "A static website hosted on Nostr"],
    ["source", "https://github.com/example/my-website"], // source code location
    ["server", "https://blossom.example.com"] // blossom server hint
  ],
  // ...other event fields
}
```

The manifest must be sent to the author's [NIP-65](65.md) `write` relays.

### Website Referencing

Host servers are expected to support a specific url that resolves to
a site manifest event address, in this form: `/.well-known/nipc8/<website identifier>`.

#### Website Identifier

The identifier for a website is a [NIP-19](#19.md) entity with some changes.

The entity encodes site manifest `d` tag, optional relay hints, and the author's pubkey using a TLV (Type-Length-Value) binary format encoded in base-62 (`[A-Za-z0-9]`) to be shorter than bech32.

To keep it even shorter than regular NIP-19 entities, we use `+` as the prefix.

Example: `+cA99KnC0UCyqHT5oI8fIkoza0jfB1lrvaWKmuh6h2EhTz2nw4R2a5qVNM`

The TLV payload contains:

| Type | Content                          | Encoding     |
|------|----------------------------------|--------------|
| 0    | `d` tag value                    | UTF-8        |
| 1    | Relay URL (optional, may repeat) | UTF-8        |
| 2    | Author pubkey                    | 32 raw bytes |

Each TLV entry is encoded as: `[type: 1 byte] [length: 1 byte] [value: length bytes]`.
The resulting binary is encoded in base-62 and prefixed with `+`.

See [reference Javascript implementation](https://github.com/44Billion/nappup/blob/main/src/helpers/nip19.js).

### Serving Files

The host server upon receiving a request on the website identifier path, downloads
the corresponding manifest. It may redirect the user to any subdomain or path mapped to the corresponding website's root.

The host server translates any request to what is avaiable on the site manifest's
`path` tags. It responds with the file downloaded from Blossom and sets the
`content-type` header related to the file's extension.

When the requested url's path doesn't end with a file extension, the server should
respond with the html file found at the `path` tag value for `<path>/index.html` and, if not available, try `<path>.html`. If nothing is found, it must fallback to the root `index.html` `path` tag.

### Nostr Apps

If the static website supports [NIP-07](07.md), it is considered a nostr web app.

When that is the case, the author should publish an `App Listing` event with
the same `d` tag from the website manifest.

The website manifest event's `created_at` field value is considered the app version.

#### App Listing Event

The app listing is an event of kind `37348`. It contains metadata for discovery and display on app marketplaces (app stores). The tags `name`, `icon` and `c` tags
are required. See commented event example:

```jsonc
{
  "kind": 37348,
  "tags": [
    ["d", "my-website"], // same as the website manifest's "d" tag
    // App name with optional ISO 639-1 language code
    ["name", "My Application", "en"],
    // The value is the blossom SHA-256 hash. Explicite mime-type.
    // May contain ISO 3166-1 alpha-2 country code if restricted to a specific audience
    ["icon", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456", "image/png", "US"],
    // "*" means app is world-wide available.
    // Else add specific ISO 3166-1 alpha-2 country codes, e.g.
    // ["c", "US"], ["c", "BR"]
    ["c", "*"],
    ["self", "<hex pubkey>"] // The app's Nostr profile.
    ["summary", "<short description>", "<?lang>"], // optional ISO 639-1 language code
    ["description", "<long description>", "?<lang>"],
    ["key-art", "<sha256>", "<mime>", "<?country>"], // optional ISO 3166-1 alpha-2 country code
    ["screenshot", "<sha256>", "<mime>", "<?country>"], // optional ISO 3166-1 alpha-2 country code
    ["screenshot", "<sha256>", "<mime>", "<?country>"],
    // Categories from a fixed list. More on that later.
    ["l", "napp.utilities:office"],
    ["l", "napp.utilities:other"]
    ["t", "notes"],
    ["t", "productivity"]
  ],
  "created_at": 1727373475
  // ...other event fields
}
```

The app listing should be sent to specialized "app listing indexer* relays such as [wss://relay.44billion.net](wss://relay.44billion.net)
and must also be sent to the author's NIP-65 write relays.

### Categories

App listings should include up to 3 `l` tags with category labels in the format `napp.<category>:<subcategory>`. The following categories and subcategories are defined:

| Category      | Subcategories |
|---------------|---------------|
| `other`       | `other` |
| `games`       | `other`, `action`, `rpg`, `strategy`, `shooter`, `fighting`, `simulation`, `puzzle`, `board`, `gambling`, `racing`, `sports`, `ar`, `vr` |
| `money`       | `other`, `crypto`, `loans`, `investments`, `wallet`, `gambling`, `raffle`, `crowdfunding`, `donation`, `jobs` |
| `shopping`    | `other`, `marketplace`, `store`, `auction` |
| `social`      | `other`, `network`, `messenger`, `blog`, `dating` |
| `audiovisual` | `other`, `podcast`, `music`, `video`, `news` |
| `utilities`   | `other`, `weather`, `office`, `finances`, `learning`, `text editor`, `image editor`, `audio editor`, `video editor`, `ar`, `vr`, `ai` |

Clients SHOULD validate category labels against this table and ignore unrecognized values.

Additional custom categorization may be achieved with `t` tags (hashtags).
