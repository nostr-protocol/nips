NIP-C4
======

Nostr Apps (napps)
------------------

`draft` `optional`

This NIP describes a method for publishing, discovering, and serving static web applications on Nostr using two complementary event kinds: an **app listing** for metadata and discoverability, and an **app bundle** for the file manifest.

Files are stored on [Blossom](https://github.com/hzrd149/blossom) servers and referenced by their SHA-256 hash, though the format is extensible to other storage backends.

### Definitions

- **App identifier (`d` tag)**: A short string (1-7 characters, `[a-z0-9]`) that uniquely identifies an app under a given pubkey and channel. **It's important to have a maximum of 7 characters.**
- **Channel**: One of `main` (stable), `next` (insider/beta), or `draft` (preview). Each channel uses a distinct event `kind` so the same `d` tag can refer to different release tracks without conflict.

### App Bundle Event

An app bundle event is an addressable event that maps filenames to their storage hashes. It serves as the file manifest for the application.

> [!TIP]
> App bundles should be uploaded to the author's NIP-65 write relays.

| Channel | Kind  |
|---------|-------|
| main    | 37448 |
| next    | 37449 |
| draft   | 37450 |

The bundle MUST use the same `d` tag as its corresponding app listing.

#### Tags

| Tag    | Format                                               | Description                       |
|--------|------------------------------------------------------|-----------------------------------|
| `d`    | `["d", "<identifier>"]`                              | Same app identifier as the listing |
| `file` | `["file", "<hash>", "<path>", "<mime>", "<service>"]`    | One per file. `hash` is the SHA-256 hex of the file. `path` is relative to the app root (e.g. `index.html`, `assets/style.css`). `mime` is the MIME type. `service` identifies the storage service. Set it to "b" if file was uploaded to Blossom servers|

#### Example

```jsonc
{
  "kind": 37448,
  "pubkey": "<32-byte-hex>",
  "content": "",
  "tags": [
    ["d", "myapp"],
    ["file", "186ea5fd14e88fd1ac49351759e7ab906fa94892002b60bf7f5a428f28ca1c99", "index.html", "text/html", "b" /* "b" is Blossom */],
    ["file", "fedcba0987654321fedcba0987654321fedcba0987654321fedcba0987654321", "assets/style.css", "text/css", "b"],
    ["file", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456", "assets/app.js", "application/javascript", "b"],
    ["file", "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef", "favicon.ico", "image/x-icon", "b"]
  ],
  "created_at": 1727373475
}
```

### App Listing Event

The app listing event is an addressable event as defined in [NIP-01](01.md). It contains metadata for discovery and display: the app's name, summary, icon, categories, and availability.

> [!TIP]
> App listings should be uploaded to specialized "app marketplace* relays such as [wss://relay.44billion.net](wss://relay.44billion.net). It should also be uploaded to the author's NIP-65 write relays.

| Channel | Kind  |
|---------|-------|
| main    | 37348 |
| next    | 37349 |
| draft   | 37350 |

#### Required tags

| Tag      | Format                                    | Description                           |
|----------|-------------------------------------------|---------------------------------------|
| `d`      | `["d", "<identifier>"]`                   | App identifier (1-7 chars, `[a-z0-9]`)|
| `name`   | `["name", "<display name>", "<lang>?"]`   | Human-readable app name. Optional ISO 639-1 language extra value in third element. |
| `icon`   | `["icon", "<sha256>", "<mime>", "<service>"]` | App icon. `sha256` is the hash of the icon blob. `mime` is the MIME type (e.g. `image/png`). `service` identifies the storage service (see [Storage services](#storage-services)). It should fallback (copy) to the favicon (e.g. favicon.ico, favicon.png etc) file if listed on the bundle event |
| `c`       | `["c", "<country or *>"]`                            | ISO 3166-1 alpha-2 uppercase country code or use a single `*` for global availability. May appear multiple times. |

Both `name` and `icon` MUST be present for a listing to be considered complete. Clients SHOULD NOT display incomplete listings in app directories.

#### Optional tags

| Tag       | Format                                          | Description                           |
|-----------|-------------------------------------------------|---------------------------------------|
| `summary` | `["summary", "<text>", "<lang>?"]`              | Short description. Optional lowercase ISO 639-1 language extra value. |
| `l`       | `["l", "napp.<category>:<subcategory>"]`        | Category label (see [Categories](#categories)). Up to 3 recommended. |
| `t`       | `["t", "<hashtag>", "<label>?"]`                | Hashtag for search/filtering. Up to 3 allowed. The hashtag value MUST be lowercased with whitespace removed. |
| `self`    | `["self", "<pubkey>"]`                             | The app's Nostr profile. |
| `auto`    | `["auto", "<field>"]`                           | Marks a field (`name`, `summary`, or `icon` etc) as auto-derived (e.g. extracted from HTML metadata). Clients updating the listing SHOULD overwrite auto-derived fields but SHOULD NOT overwrite manually-set fields. |
| `description`| `["description", "<text>", "<lang>?"]`       | Long description. Optional ISO 639-1 language extra value. |
| `key-art`| `["key-art","<sha256>", "<mime>", "<service>"]`       | Main image e.g. to show on app card |
| `screenshot`| `["screenshot", "<sha256>", "<mime>", "<service>"]`       | Screenshot image |


#### Example

```jsonc
{
  "kind": 37348,
  "pubkey": "<32-byte-hex>",
  "content": "",
  "tags": [
    ["d", "myapp"],
    ["name", "My Application"],
    ["summary", "A decentralized note-taking app"],
    ["icon", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456", "image/png", "b"],
    ["c", "*"],
    ["l", "napp.utilities:office"],
    ["t", "notes"],
    ["t", "productivity"],
    ["auto", "summary"]
  ],
  "created_at": 1727373475
}
```

### Storage services

The `service` element in `file` and `icon` tags identifies how to retrieve the blob for a given hash. This NIP defines one service and reserves an identifier for a second:

| Service ID | Name    | Description |
|------------|---------|-------------|
| `b`        | Blossom | File is stored on [Blossom](https://github.com/hzrd149/blossom) servers and retrievable via [BUD-01](https://github.com/hzrd149/blossom/blob/master/buds/01.md) `GET /<sha256>`. |
| `i`        | IRFS    | Reserved. File is stored on Nostr relays (unreleased spec; a fallback when user has no Blossom servers set up - out of scope) |

Besides IRFS, future storage services additions are not expected to avoid hurting interoperability.
It is recommended to upload all files to the same service type.

### Categories

App listings MAY include up to 3 `l` tags with category labels in the format `napp.<category>:<subcategory>`. The following categories and subcategories are defined:

| Category      | Subcategories |
|---------------|---------------|
| `other`       | `other` |
| `games`       | `other`, `action`, `rpg`, `strategy`, `shooter`, `fighting`, `simulation`, `puzzle`, `board`, `gambling`, `racing`, `sports`, `ar`, `vr` |
| `money`       | `other`, `crypto`, `loans`, `investments`, `wallet`, `gambling`, `raffle`, `crowdfunding`, `donation`, `jobs` |
| `shopping`    | `other`, `marketplace`, `store`, `auction` |
| `social`      | `other`, `network`, `messenger`, `blog`, `dating` |
| `audiovisual` | `other`, `podcast`, `music`, `video`, `news` |
| `utilities`   | `other`, `weather`, `office`, `finances`, `learning`, `text editor`, `image editor`, `audio editor`, `video editor`, `ar`, `vr`, `ai` |

Clients SHOULD validate category labels against this table and ignore unrecognized values.

### App identifier constraints

The `d` tag value (app identifier) MUST satisfy:

- Length: 1 to 7 characters
- Character set: `[a-z0-9]` (lowercase alphanumeric, i.e. base-36)

Publishers that need to derive an identifier from an arbitrary string (e.g. a folder name) SHOULD hash the string with SHA-1 and encode the first 4 bytes in base-36, truncated to 7 characters.

### NIP-19 `napp` entity encoding

Apps are optionally referenced using a `napp` entity string (that does NOT start with a `napp1` prefix but with `+`, see table below). See [reference Javascript implementation](https://github.com/44Billion/nappup/blob/main/src/helpers/nip19.js).

> [!NOTE]
> You can use this to reference a Nostr app e.g. as a path `https://host-server-example.com/<napp entity>` or
> inside a kind:1 note's content as `nostr:<napp entity>`

The entity encodes app identifier, optional relay hints, and the author's pubkey using a TLV (Type-Length-Value) binary format encoded in base-62 to be smaller than bech32.

The channel is indicated by a prefix of `+` characters:

| Prefix | Channel |
|--------|---------|
| `+`    | main    |
| `++`   | next    |
| `+++`  | draft   |

The TLV payload contains:

| Type | Content          | Encoding |
|------|------------------|----------|
| 0    | App identifier   | UTF-8    |
| 1    | Relay URL (optional, may repeat) | UTF-8 |
| 2    | Author pubkey    | 32 raw bytes |

Each TLV entry is encoded as: `[type: 1 byte] [length: 1 byte] [value: length bytes]`.

The resulting binary is encoded in base-62 and prefixed with the channel prefix. For example, a stable app might be referenced as `+5kJx8...` while its insider channel would be `++5kJx8...`.

The entity format matches the regex pattern: `^\+{1,3}[A-Za-z0-9]{48,}$`

### Publishing flow

A publisher SHOULD follow this sequence:

1. **Discover relays**: Fetch the author's [NIP-65](65.md) relay list (kind `10002`) to determine write relays.
2. **Discover blossom servers**: Fetch the author's [BUD-03](https://github.com/hzrd149/blossom/blob/master/buds/03.md) server list (kind `10063`) and health-check each server.
3. **Upload files**: Upload all application files to one or more healthy blossom servers. Compute the SHA-256 hash for each file.
4. **Upload icon**: If an icon is provided separately (not part of the app's file tree), upload it to blossom servers.
5. **Publish app listing**: Create and sign an app listing event (kind 373xx) with metadata tags. If a previous listing exists and only auto-derived fields have changed, update only those fields.
6. **Publish app bundle**: Create and sign an app bundle event (kind 374xx) with `file` tags mapping each filename to its SHA-256 hash. If the file manifest is identical to the most recent bundle, skip republishing.

When updating an existing app, the `created_at` of the new event MUST be greater than the previous event's `created_at`. If the current timestamp is not greater, the publisher SHOULD use `previous.created_at + 1`. The app version is the bundle event's `created_at` field value.

### Host server implementation

A host server is an HTTP server that resolves and serves napp content to web browsers.

#### Resolving apps

Host servers may resolve a pubkey and app identifier however they see fit (e.g. using encoded pathnames or subdomains). When receiving a request, the host server MUST:

1. Determine the author pubkey and app identifier from the request.
2. Fetch the author's [NIP-65](65.md) relay list.
3. Query for the app bundle event at the author's write relays:

```jsonc
{ "kinds": [37448], "authors": ["<pubkey>"], "#d": ["<identifier>"] }
```

4. Extract `file` tags from the bundle to build a path-to-hash mapping.

#### Resolving files

Given a request path, the host server MUST:

1. Find the `file` tag whose path element matches the request. If the request path does not include a filename (e.g. `/` or `/blog/`), append `index.html`.
2. Use the SHA-256 hash from the matching `file` tag to retrieve the blob:
   - If the storage service is `b` (Blossom): check the author's kind `10063` blossom server list and retrieve via `GET /<sha256>`.
3. Serve the blob with appropriate `Content-Type` derived from the `mime` element of the `file` tag.

#### Handling not found

If no matching `file` tag exists for the requested path, the host server SHOULD look for a file tagged as `404.html` and serve it with HTTP status 404. If no `404.html` exists, the server MUST return a generic 404 response.

#### Subdomain-based app URLs

To enable multiple interoperable gateway servers, this NIP defines a canonical subdomain format for serving napps. An app is accessed at:

```
https://<subdomain>.<gateway-domain>
```

The subdomain encodes the channel, the app author's pubkey, and the app identifier. Its structure is:

```
nostr<channel><pubkeyB36><dTag>
```

| Segment       | Length    | Charset      | Description |
|---------------|-----------|--------------|-------------|
| `userSlug`    | 5 chars   | `[a-z0-9]`   | Set always to "nostr" placeholder unless you're a specialized gateway that encodes the logged-in user pubkey here. (out of scope) |
| `channel`     | 1 char    | `[abc]`      | `a` = main (kind 37448), `b` = next (kind 37449), `c` = draft (kind 37450) |
| `pubkeyB36`   | 50 chars  | `[a-z0-9]`   | The app author's pubkey encoded in base-36, zero-padded to 50 characters |
| `dTag`        | 1-7 chars | `[a-z0-9]`   | The app identifier (`d` tag value) |

**Total length: 57-63 characters**, fitting within the 63-character DNS label limit (RFC 1035).

The subdomain MUST match the regex:

```
^nostr[abc][a-z0-9]{50}[a-z0-9]{1,7}$
```

##### Example

Given:
- Channel: main
- App author pubkey (hex): `7e7e9c42a91bfef19fa929e5fda1b72e0ebc1a4c1141673e2794234d86addf4e`
- App identifier: `44b`

Steps:
1. Channel main -> 'a'
2. Author pubkey -> base-36 (50 chars): `35hya087sv94zz8ba0au26r7o7wodxktvjqkata0j1rfuoxj66`
3. Subdomain: `nostr` + `a` + `35hya087sv94zz8ba0au26r7o7wodxktvjqkata0j1rfuoxj66` + `44b`

Result: `nostra35hya087sv94zz8ba0au26r7o7wodxktvjqkata0j1rfuoxj6644b`

Full URL: `https://nostra35hya087sv94zz8ba0au26r7o7wodxktvjqkata0j1rfuoxj6644b.napp.example/`

##### Parsing a subdomain

To extract components from a subdomain, match against:

```
^nostr(?<channel>[abc])(?<pubkeyB36>[a-z0-9]{50})(?<dTag>[a-z0-9]{1,7})$
```

To recover the author's hex pubkey from `pubkeyB36`, decode the base-36 string to a big-endian unsigned integer and encode as 32 bytes (64 hex characters).

### App configuration file

Publishers MAY include a `.well-known/napp.json` file in their application directory. This file is consumed by the publishing tool to populate listing metadata and is NOT included in the published bundle.

The file MAY contain the following fields:

```jsonc
{
  "stallName": [["My App", "en"]],       // [value, ISO 639-1 lang?]
  "stallSummary": [["Short description", "en"]],
  "stallDescription": [["Long description", "en"]],
  "stallIcon": [["data:image/png;base64,...", "US"], ["data:image/png;base64,..."]],  // data URL, optionally add the target country
  "self": [["https://myapp.com"]],
  "country": ["US", "GB"],               // ISO 3166-1 alpha-2 codes or ["*"] (the default for global availability)
  "category": [["utilities", ["office", "ai"]]],  // [category, [subcategories]]
  "hashtag": [["productivity", "Productivity"]],    // [tag, display label?]
  "keyArt": [["data:image/png;base64,...", "JP"], ["data:image/png;base64,..."]], // e.g. Cartridge art for a game
  "screenshot": [["data:image/png;base64,..."], ["data:image/png;base64,...", "BR"]]
}
```

### Deduplication and updates

Since app listing and app bundle events are addressable events (NIP-01), relays MUST store only the most recent version (highest `created_at`) per `kind + pubkey + d` combination. Publishers SHOULD:

- Compare the current file manifest against the most recent bundle before publishing. If identical, skip the bundle update to avoid bumping the app version (which is the bundle's `created_at` field).
- Preserve manually-set metadata fields when auto-deriving values from HTML content.
- Re-upload existing events to relays that are missing them, rather than creating new events with identical content.
