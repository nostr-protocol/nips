NIP-XX
======

Key Derivation from LNURL-auth via Split-Knowledge Nonce (SplitSig)
--------------------------------------------------------------------

`draft` `optional`

This NIP describes a method for deriving a Nostr signing keypair from an LNURL-auth interaction, without the authenticating server being able to derive the private key.

LNURL-auth (LUD-04) lets users sign into web services by scanning a QR code with their Lightning wallet — no username, no password. The wallet signs a challenge with a domain-specific key derived from its BIP-32 seed, proving identity without exposing the seed.

Motivation
----------

Users want to log into Nostr web clients with their Lightning wallet. Existing approaches have tradeoffs:

- **NIP-06** (mnemonic): requires the user to enter or store a seed phrase separately from their wallet.
- **NIP-07** (browser extension): requires installing an extension.
- **NIP-46** (remote signer): requires running a separate signer service.
- **Pasting nsec**: exposes the private key to the web client's server.

LNURL-auth (LUD-04) provides authentication via a Lightning wallet, but the ECDSA signature passes through the server. Deriving a key directly from that signature (as done in Ethereum by zkSync and EIP-2645) would give the server the private key.

SplitSig adds a client-side nonce so neither the signature alone nor the nonce alone can derive the key.

Specification
-------------

The Nostr private key is derived as:

    nsec_bytes = SHA256( ecdsa_signature || nonce )

Where:

- `ecdsa_signature` is the raw DER-encoded bytes of the ECDSA signature from the LNURL-auth callback, produced by the wallet signing the `k1` challenge with its linking key (LUD-04, RFC 6979). The signature is transmitted as a hex string; implementations MUST decode the hex to raw bytes before hashing.
- `nonce` is a 256-bit random value generated by the client via `crypto.getRandomValues()`, never transmitted to the server. Also decoded from hex to raw bytes before hashing.
- `||` denotes byte concatenation of raw bytes (not hex strings).

The corresponding Nostr public key (npub) is derived normally:

    nsec = nsec_bytes  (as 32-byte hex, convertible to bech32 nsec)
    npub = nsec * G    (secp256k1 scalar multiplication, x-only)

### Key Derivation Flow

1. Client generates a random 32-byte `nonce` and stores it locally.
2. Client initiates LNURL-auth with the server (LUD-04).
3. Wallet signs the `k1` challenge and sends `sig` + `key` to the server callback via GET (per LUD-04).
4. Client retrieves the `sig` from the server's auth status endpoint (see [Server Requirements](#server-requirements) below).
5. Client decodes the hex-encoded `sig` to raw DER bytes.
6. Client computes `nsec = SHA256(sig_bytes || nonce_bytes)`.
7. Client computes `npub` from `nsec`.
8. Client sends only the `npub` to the server.
9. Client stores the `nonce` in a recovery kit for future re-derivation.

### Server Requirements

Standard LNURL-auth (LUD-04) server implementations verify the ECDSA signature and create a session, but do **not** expose the raw signature back to the browser. SplitSig requires the server to provide the signature to the client. This is a non-standard extension.

The server MUST:

1. **Store the raw ECDSA signature** (hex-encoded DER) after successful verification of the LNURL-auth callback.
2. **Expose the signature** via a status endpoint that the browser polls.

The status endpoint MUST return a JSON response with at minimum:

```json
{
  "verified": true,
  "signature": "<hex-encoded DER ECDSA signature>",
  "session_token": "<session token>"
}
```

The `signature` field MUST contain the exact hex string the wallet sent in the callback `sig` parameter. The server MUST NOT modify or re-encode the signature.

The signature MAY be returned only once and cleared from server memory after delivery to the client, since the server does not need it after authentication.

### Deterministic k1

To ensure the same wallet always produces the same signature for the same context, the `k1` challenge MUST be deterministic:

    k1 = SHA256( prefix || 0x00 || context_identifier )

Where:

- `prefix` is an application-defined string identifying the service (e.g. `"splitsig"`, `"ark-escrow"`, `"myapp"`). Each application SHOULD use a fixed prefix.
- `0x00` is a null byte separator to prevent collisions between prefix and context.
- `context_identifier` is an application-defined string (e.g. a user ID, a deal ID, or the string `"nostr"`). If multiple fields are concatenated, they MUST be separated by `0x00`.

A random k1 (as used in standard LNURL-auth) MUST NOT be used, because the same wallet would produce a different signature each time, making recovery impossible.

### Recovery

To re-derive the same key on a new device:

1. Load the recovery kit (contains the `nonce` and context parameters).
2. Authenticate via LNURL-auth with the same Lightning wallet against the same domain.
3. The wallet produces the same deterministic signature (RFC 6979: same key + same `k1` = same `sig`).
4. Client computes `SHA256(same sig || same nonce)` = same `nsec`.

The LNURL-auth linking key is derived from the wallet's BIP-32 seed at path `m/138'/...` (LUD-04). It is stable across app updates, OS upgrades, and device changes.

### Recovery Kit Format

The recovery kit is a JSON object the user downloads containing the nonce and context parameters needed to re-derive the key. It is the user's backup — equivalent to writing down a seed phrase.

Required fields:

```json
{
  "splitsig_version": 1,
  "nonce": "<256-bit hex>",
  "auth_domain": "<LNURL-auth domain>",
  "context_identifier": "<application-defined context string>",
  "linking_pubkey": "<compressed 33-byte hex, optional — for verification>"
}
```

- `nonce`: the 256-bit random value, hex-encoded. This is the secret piece.
- `auth_domain`: the domain used for LNURL-auth. The wallet derives its linking key from this domain (LUD-04). Needed to re-derive the same linking key on a new device.
- `context_identifier`: the value used in the deterministic k1 computation. Needed to reconstruct the same k1 challenge.
- `linking_pubkey`: the wallet's compressed public key from the LNURL-auth callback. Optional but recommended for verifying the correct wallet is used during recovery.

Applications MAY embed additional fields (e.g. escrow parameters, role identifiers). The `splitsig_version` field allows future format changes.

Security Properties
-------------------

### Knowledge Distribution

| Piece             | Server  | Client  | Recovery Kit |
|-------------------|---------|---------|--------------|
| ECDSA signature   | **yes** | yes     | no           |
| Nonce             | **no**  | yes     | **yes**      |
| Private key (nsec)| **no**  | yes     | no (derived) |
| Public key (npub) | **yes** | yes     | no           |

### Threat Model

- **Compromised server**: has the signature but not the nonce. Cannot derive `nsec`. Brute-forcing a 256-bit nonce is computationally infeasible.
- **Compromised client**: has both pieces. Can derive `nsec`. This is the trusted code delivery problem, inherent to all web applications.
- **Stolen recovery kit**: has the nonce but not the signature. Cannot derive `nsec` without the wallet's BIP-32 seed.
- **Lost wallet seed**: different seed = different linking key = different signature = different `nsec`.
- **Lost recovery kit**: `nonce` is lost. `nsec` cannot be re-derived.

### Implementation Notes

- The derived private key SHOULD be stored in ephemeral storage (e.g. `sessionStorage`) and cleared when the session ends. It SHOULD NOT be persisted to disk or `localStorage`.
- The nonce SHOULD be persisted in `localStorage` (or equivalent) to survive page reloads, and included in the downloadable recovery kit.
- ECDSA signatures in LNURL-auth use secp256k1, the same curve as Nostr. However, LNURL-auth uses ECDSA while Nostr uses Schnorr (BIP-340). The SplitSig derivation produces a raw 32-byte private key that is valid for both signature schemes.

Cryptographic Justification
----------------------------

SHA256 of an ECDSA signature produces a uniformly distributed 256-bit value suitable as a secp256k1 private key. This technique is used in production by zkSync (Matter Labs), Umbra Protocol, and StarkWare (EIP-2645). Alex Gluchowski (Matter Labs) notes: "SHA256 is important from the cryptanalytic perspective: signatures are elliptic curve points, so we need to break any potential relation between the two keys."

The nonce adds 256 bits of entropy independent of the signature, ensuring the derived key cannot be computed from the signature alone.

Reference Implementation
-------------------------

- Library: [github.com/Antisys/splitsig](https://github.com/Antisys/splitsig)
- Install: `npm install github:Antisys/splitsig`

```javascript
import { generateNonce, deriveKey } from '@antisys/splitsig';

const nonce = generateNonce();
const key = deriveKey(signatureHex, nonce);
// key.privateKey     → nsec (32-byte hex)
// key.publicKeyXOnly → npub (32-byte x-only hex)
```

Prior Art
---------

- **NIP-06**: derives Nostr key from BIP-39 mnemonic at `m/44'/1237'/0'/0/0`. Requires seed phrase access.
- **LUD-04**: LNURL-auth specification. Defines the linking key derivation from wallet seed.
- **LUD-13**: derives LNURL-auth key from `signMessage`. The `hashingKey` is a valid secp256k1 key but is per-domain and known to the server.
- **EIP-2645**: derives StarkWare keys from Ethereum signatures. No nonce split.
- **zkSync**: `SHA256(sign(msg))` for L2 key derivation. No nonce split.
