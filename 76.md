NIP-76
======

Speedrunning Leaderboard Entry
------------------------------

`draft` `optional`

## Abstract

This NIP defines event kind `7602` as a submission to a game's speedrunning leaderboard. It enables cross-platform, decentralized tracking of speedruns with a high degree of metadata granularity, allowing runners to publish their achievements and proof directly to relays.

## Event Format

An event of kind `7602` represents a speedrunning submission. It must contain a `g` tag specifying the game being run. 

Each game's leaderboard may use optional `c` (Category), `v` (Variation), `plt` (Platform), `lvl` (Level), `reg` (Region), and `ver` (Version) tags to allow for the creation of alternate sub-leaderboards under one game.

```jsonc
{
  "kind": 7602,
  "tags": [
    ["g", "<Unique Identifier for the game>"],
    ["c", "<Category Identifier the submission is for>", "<Long Name>"], // e.g., A%G (Any% Glitchless), AlAc (All Achievements)
    ["v", "<Variation of the category the submission is for>", "<Long Name>"], // e.g., for Celeste AnyP (Any%), ARB (All Red berries), TE (True Ending) etc.
    ["plt", "<Platform Identifier>"], // e.g., N64, EMU
    ["reg", "<3 Letter ISO Country Code>"], // e.g., JPN, AUS
    ["lvl", "<Level>"], // e.g., FG, W1L1, F3OS
    ["ver", "<Version>"], // Version the submission is ran on e.g., 1.4.0.0
    ["rlt", "<Duration>"], // Real Life Time (decimals allowed for millisecond precision, e.g., 5781.450)
    ["igt", "<Duration>"], // In-Game Time (decimals allowed for millisecond precision)
    ["p", "<pubkey>"], // Optional: Pubkey of co-op runners. Can be included multiple times for multiplayer runs
    ["proof", "<Link to proof>"] // e.g., Twitch VOD, replay file
  ],
  "content": "<Description for run>" // User written comment about run if desired
}
```

Tags may be entered multiple times if required, such as a game having multiple varieties under one category. E.g., Pizza Tower Full Game Any% has "Playable Character" and "Glitches?" varieties.

For multiplayer or co-op runs, multiple `p` tags should be included to appropriately credit all runners involved.

An example of a potential Super Mario 64 120 Star N64 run submission:

## Client Behavior and Comments

[NIP-22](22.md) comments should be used to discuss and comment on runs. This allows the community to interact with the runner, verify proof, and point out discrepancies.

```jsonc
{
  "kind": 1111,
  "tags": [
    ["e", <eventId>, <relayUrl>, "reply"],
    ["p", <pubkey>]
  ],
  "content": "<Comment on the run>",
  ...
}
```

Clients may decide whether or not to display a run based on user comments, letting the community organically figure out if a run is legitimate or faked.

## Game Identifiers (Sub-NIP Proposal)

Currently, identifying a game globally in a decentralized manner poses a challenge (e.g., differentiating between regions, re-releases, and similar titles without relying on centralized stores like Steam or IGDB). 

To solve this, a Nostr-native **Game Dictionary** is proposed:

1. **Event Kind**: A parameterized replaceable event (e.g., `kind: 30076`) should be used to define a Game Dictionary entry.
2. **Identifier (`g` tag)**: The `g` tag must be a unique, standardized short-code or abbreviation representing the game.
3. **Metadata Tags**: The event must include tags providing comprehensive metadata about the game, such as:
   - `["name", "<Full Game Title>"]`
   - `["developer", "<Developer Name>"]`
   - `["release", "<Year>"]`
   - `["image", "<URL to Box Art>"]`
   - `["alt_id", "<External Reference>", "<Optional Source/Namespace>"]` // e.g., `["alt_id", "945360", "steam"]` or `["alt_id", "SLUS-00892", "ps1"]`

When a user submits a run using `kind: 7602`, the `g` tag must reference the exact `g` tag identifier established in this Game Dictionary. This ensures all clients group runs for the exact same game accurately, facilitating interoperable leaderboards.
