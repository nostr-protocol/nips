NIP-5B
======

Embeddable Nostr Web Apps
-------------------------

`draft` `optional`

This NIP describes a method for advertising [NIP-5A](5A.md) static websites as embeddable web apps.

These are purely client-side app made of web assets (HTML, CSS, JS) that can be loaded dynamically from inside higher-order apps. Those higher-other apps will ensure the embedded app has access to [NIP-07](07.md) `window.nostr` APIs.

### App Listing Event

The app listing is an event of kind `37348`. It contains metadata for discovery and display on app marketplaces (app stores). The tags `d`, `name`, `icon` and `c` are required. See commented event example:

```jsonc
{
  "kind": 37348,
  "tags": [
    ["d", "my-website"], // same as the website manifest's "d" tag
    // App name with optional ISO 639-1 language code
    ["name", "My Application", "en"],
    // The value is the blossom SHA-256 hash. Explicit MIME type.
    // May contain ISO 3166-1 alpha-2 country code if restricted to a specific audience
    ["icon", "a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456", "image/png", "US"],
    // "*" means the app is available worldwide.
    // Else add specific ISO 3166-1 alpha-2 country codes, e.g.
    // ["c", "US"], ["c", "BR"]
    ["c", "*"],
    ["self", "<hex pubkey>"], // The app's Nostr profile.
    ["summary", "<short description>", "<?lang>"], // optional ISO 639-1 language code
    ["description", "<long description>", "?<lang>"],
    ["key_art", "<sha256>", "<mime>", "<?country>"], // optional ISO 3166-1 alpha-2 country code
    ["screenshot", "<sha256>", "<mime>", "<?country>"], // optional ISO 3166-1 alpha-2 country code
    ["screenshot", "<sha256>", "<mime>", "<?country>"],
    // Categories from a fixed list. More on that later.
    ["l", "napp.utilities:office"],
    ["l", "napp.utilities:other"],
    ["t", "notes"],
    ["t", "productivity"]
  ],
  "created_at": 1727373475
  // ...other event fields
}
```

The app listing should be sent to specialized "App Listing Indexer" relays such as [wss://relay.44billion.net](wss://relay.44billion.net)
and must also be sent to the author's NIP-65 `write` relays.

#### App Categories

App listings should include up to 3 `l` tags with category labels in the format `napp.<category>:<subcategory>`. The following categories and subcategories are defined:

| Category      | Subcategories |
|---------------|---------------|
| `other`       | `other` |
| `games`       | `other`, `action`, `rpg`, `strategy`, `shooter`, `fighting`, `simulation`, `puzzle`, `board`, `gambling`, `racing`, `sports`, `ar`, `vr` |
| `money`       | `other`, `crypto`, `loans`, `investments`, `wallet`, `gambling`, `raffle`, `crowdfunding`, `donation`, `jobs` |
| `shopping`    | `other`, `marketplace`, `store`, `auction` |
| `social`      | `other`, `network`, `messenger`, `blog`, `dating` |
| `audiovisual` | `other`, `podcast`, `music`, `video`, `news` |
| `utilities`   | `other`, `weather`, `office`, `finances`, `learning`, `text editor`, `image editor`, `audio editor`, `video editor`, `ar`, `vr`, `ai` |

Clients SHOULD validate category labels against this table and ignore unrecognized values.

Additional custom categorization may be achieved with `t` tags (hashtags).

### App Identifier

The identifier for a website is a [NIP-19](19.md) entity with some changes.

The entity encodes the site manifest's `d` tag, optional relay hints, and the author's pubkey using a TLV (Type-Length-Value) binary format encoded in base-62 (`[A-Za-z0-9]`) to make it shorter than bech32.

To keep it even shorter than regular NIP-19 entities, we use `+` as the prefix.

Example: `+cA99KnC0UCyqHT5oI8fIkoza0jfB1lrvaWKmuh6h2EhTz2nw4R2a5qVNM`

The TLV payload contains:

| Type | Content                          | Encoding     |
|------|----------------------------------|--------------|
| 0    | `d` tag value                    | UTF-8        |
| 1    | Relay URL (optional, may repeat) | UTF-8        |
| 2    | Author pubkey                    | 32 raw bytes |

Each TLV entry is encoded as: `[type: 1 byte] [length: 1 byte] [value: length bytes]`.
The resulting binary is encoded in base-62 and prefixed with `+`.

See [reference Javascript implementation](https://github.com/44Billion/nappup/blob/main/src/helpers/nip19.js).
