NIP-95
======

Hybrid Peer-to-Peer Relay Protocol
-----------------------------------

`draft` `optional` `author:lucianocasalunga`

This NIP defines a protocol extension that enables Nostr relays to operate in a hybrid peer-to-peer mode, where connected clients can share cached events directly with each other via WebRTC, reducing relay load and improving decentralization.

## Motivation

Current Nostr relays operate in a pure client-server model. Every event request must be served by the relay, creating a single point of failure and a scalability bottleneck. If a relay goes offline, all connected clients lose access to events.

Inspired by Spotify's P2P music distribution model, this NIP proposes a hybrid architecture where:

- The relay acts as a **Seed Node** (authoritative source + peer coordinator)
- Stable clients become **Super Peers** (cache and share recent events via P2P)
- Other clients are **Casual Peers** (consume from Super Peers with relay fallback)

## Architecture

### Three-Layer Model

```
┌─────────────────────────────────────────────┐
│              RELAY SEED NODE                │
│  (strfry/nostr-rs-relay + Signaling Server) │
│  - Stores ALL events permanently            │
│  - Validates signatures + write policies    │
│  - Coordinates peer discovery (ICE/STUN)    │
│  - Serves historical events (>24h)          │
└──────────────┬──────────────────────────────┘
               │ WebSocket (NIP-01)
       ┌───────┴───────┐
       │               │
┌──────▼──────┐ ┌──────▼──────┐
│ SUPER PEER  │◄─►│ SUPER PEER  │  WebRTC Data Channel
│ (stable     │ │ (stable     │
│  client)    │ │  client)    │
│ Cache: 24h  │ │ Cache: 24h  │
└──────┬──────┘ └──────┬──────┘
       │ WebRTC         │ WebRTC
┌──────▼──────┐ ┌──────▼──────┐
│CASUAL PEER  │ │CASUAL PEER  │
│ (mobile/    │ │ (temporary  │
│  unstable)  │ │  client)    │
└─────────────┘ └─────────────┘
```

### Peer Classification

**Super Peer** criteria:
- Connected for >30 minutes continuously
- Available bandwidth >5 Mbps
- Local storage available >100 MB
- Latency to seed <100ms

**Casual Peer** (default):
- Any client that doesn't meet Super Peer criteria
- Mobile clients (battery conservation)
- Unstable connections

## Protocol Flow

### New Message Types

This NIP introduces the following new message types sent between clients and relays:

#### Client to Relay

- `["PEER_REGISTER", <capabilities_json>]` — Register as potential Super Peer
- `["PEER_REQUEST", <subscription_id>, <filter>]` — Request events with P2P preference
- `["PEER_STATS", <stats_json>]` — Report P2P sharing statistics

#### Relay to Client

- `["PEER_PROMOTED", <peer_id>, <ice_config>]` — Client promoted to Super Peer
- `["PEER_OFFER", <subscription_id>, <peer_list>]` — List of Super Peers that have requested events
- `["PEER_DEMOTED", <reason>]` — Client demoted to Casual Peer

### Flow 1: Publishing an Event

```
1. Client → Relay (WebSocket): ["EVENT", <event>]
2. Relay: Validates signature + write policy
3. Relay: Stores in database
4. Relay → Super Peers (WebSocket): ["EVENT", <event>]
5. Super Peers: Cache event locally (IndexedDB, max 24h)
6. Super Peers → Casual Peers (WebRTC): Share event P2P
```

### Flow 2: Requesting Recent Events (<24h)

```
1. Client → Relay: ["PEER_REQUEST", <sub_id>, <filter>]
2. Relay: Identifies Super Peers with matching cached events
3. Relay → Client: ["PEER_OFFER", <sub_id>, [{peer_id, ice_candidates}, ...]]
4. Client ↔ Super Peer: Establish WebRTC Data Channel
5. Super Peer → Client: Send events via P2P
6. FALLBACK: If P2P fails within 5s, Relay sends via WebSocket
```

### Flow 3: Requesting Historical Events (>24h)

```
1. Client → Relay: ["REQ", <sub_id>, <filter>]  (standard NIP-01)
2. Relay: Query database
3. Relay → Client: ["EVENT", ...] via WebSocket (no P2P)
```

## WebRTC Signaling

The relay acts as a WebRTC signaling server. ICE candidates are exchanged through the existing WebSocket connection:

```json
// Relay sends ICE configuration to promoted Super Peer
["PEER_PROMOTED", "peer_abc123", {
  "iceServers": [
    {"urls": "stun:stun.l.google.com:19302"},
    {"urls": "turn:turn.libernet.app:3478", "username": "...", "credential": "..."}
  ],
  "maxPeers": 10,
  "cacheWindow": 86400
}]
```

```json
// Relay offers peer list for event retrieval
["PEER_OFFER", "sub_123", [
  {
    "peerId": "peer_abc123",
    "iceCandidates": [...],
    "cachedKinds": [1, 6, 7],
    "eventCount": 342,
    "avgLatency": 45
  }
]]
```

## Security Considerations

### Event Integrity
- Clients MUST validate Schnorr signatures (NIP-01) on ALL events received via P2P
- Events with invalid signatures MUST be rejected and the sending peer reported
- The relay maintains a peer reputation system; peers serving invalid events are demoted

### Privacy
- P2P connections expose IP addresses between peers
- Relays SHOULD offer a "blind relay mode" where the relay proxies WebRTC (privacy at the cost of latency)
- Clients MUST opt-in to P2P mode (not enabled by default)

### Abuse Prevention
- Super Peers are rate-limited (max events shared per minute)
- Relay monitors P2P statistics for anomalies
- Malicious peers are banned progressively (1h → 24h → permanent)

## Relay Information Extension

Relays supporting NIP-95 SHOULD advertise this in their NIP-11 relay information document:

```json
{
  "supported_nips": [1, 11, 95],
  "limitation": {
    "p2p_enabled": true,
    "p2p_max_peers": 50,
    "p2p_cache_window": 86400,
    "p2p_min_bandwidth": 5000000
  }
}
```

## Client Implementation Notes

- Clients SHOULD use the [WebRTC API](https://developer.mozilla.org/en-US/docs/Web/API/WebRTC_API) for P2P connections
- Local event cache SHOULD use IndexedDB with a 24-hour TTL
- P2P mode MUST be opt-in with clear user consent
- Clients SHOULD display P2P connection status (e.g., "Connected via 3 peers")
- Mobile clients SHOULD default to Casual Peer mode to conserve battery

## Backwards Compatibility

This NIP is fully backwards compatible with existing relays and clients:

- Relays that don't support NIP-95 simply ignore `PEER_*` messages
- Clients that don't support NIP-95 use standard `REQ`/`EVENT` flow (NIP-01)
- The P2P layer is an optional enhancement, not a requirement
