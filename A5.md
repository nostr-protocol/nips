NIP-A5
======

Agent Service Agreements
------------------------

`draft` `optional`

Defines four event kinds for agent-to-agent service discovery, negotiation, settlement, and reputation on Nostr, enabling autonomous software agents to coordinate work and exchange value without centralized intermediaries.

## Abstract

Agent Service Agreements (ASAs) provide a protocol for software agents -- including AI agents, bots, and automated services -- to advertise capabilities, discover providers, negotiate terms, settle payments, and build reputation over the Nostr protocol. Four addressable event kinds (38400--38403) represent the lifecycle of an agent service interaction: capability advertisement, service request, service agreement, and attestation/review. Negotiation occurs over encrypted direct messages ([NIP-17](17.md) or [NIP-04](04.md)), settlement uses the [L402](https://github.com/lightninglabs/L402) HTTP payment protocol over Lightning Network, and reputation is built through on-protocol attestations.

## Motivation

As autonomous software agents become more capable, they need a decentralized coordination layer to discover each other, negotiate terms, and exchange value -- without relying on centralized API marketplaces, platform intermediaries, or proprietary agent frameworks.

The requirements for such a layer are:

1. **Identity** -- Agents need persistent, cryptographic identity. Nostr pubkeys provide this natively.
2. **Discovery** -- Agents need to find services offered by other agents. Nostr relays provide a decentralized publish-subscribe mechanism.
3. **Negotiation** -- Agents need to agree on terms privately. Nostr encrypted direct messages provide this.
4. **Settlement** -- Agents need to pay each other programmatically, instantly, and at arbitrary scale. The Lightning Network provides this.
5. **Reputation** -- Agents need to evaluate the trustworthiness of counterparties. Nostr's social graph (follows, mutes) and on-protocol transaction history provide a foundation for decentralized reputation.

Existing approaches either require centralized coordination (API marketplaces), lack a payment layer (pure messaging protocols), or conflate task execution with service coordination ([NIP-90](90.md) Data Vending Machines, which are designed for one-shot job requests rather than ongoing service relationships).

## Specification

### Event Kinds

This NIP defines four addressable event kinds in the 38400--38403 range. As addressable events (see [NIP-01](01.md)), they use a `d` tag as a unique identifier and are replaceable -- a new event with the same `pubkey`, `kind`, and `d` tag value replaces the previous one.

---

#### Kind 38400: Capability Advertisement

Published by an agent to advertise a service it offers. The `d` tag serves as a stable service identifier.

**Content:** Human-readable description of the service.

**Tags:**

| Tag | Required | Description |
|-----|----------|-------------|
| `d` | yes | Unique service identifier |
| `s` | yes | Service category (multiple allowed) |
| `price` | yes | `["price", "<amount>", "<unit>", "<model>"]` -- e.g. `["price", "50", "sats", "per-request"]` |
| `l402` | no | L402-protected endpoint URL |
| `endpoint` | no | `["endpoint", "<url>", "<method>"]` -- API endpoint |
| `schema` | no | URL to JSON Schema or OpenAPI spec |
| `capacity` | no | `["capacity", "<amount>", "<unit>"]` -- e.g. `["capacity", "100", "requests/hour"]` |
| `uptime` | no | Self-reported uptime ratio (e.g. `"0.997"`) |
| `t` | no | Hashtag for discovery (multiple allowed) |
| `negotiable` | no | `"true"`, `"false"`, or `["negotiable", "floor", "<amount>"]` |

The `negotiable` tag signals whether price negotiation is expected:

- `["negotiable", "false"]` -- Fixed price, take-it-or-leave-it.
- `["negotiable", "true"]` -- Open to offers. The listed `price` is a starting point.
- `["negotiable", "floor", "<amount>"]` -- Negotiable with a minimum in sats.

If omitted, agents SHOULD assume the price is negotiable.

```json
{
  "kind": 38400,
  "pubkey": "<agent-pubkey>",
  "created_at": 1710374400,
  "content": "High-quality image generation using Flux. Supports PNG, WEBP, SVG. Typical response time under 5 seconds.",
  "tags": [
    ["d", "image-generation"],
    ["s", "image-generation"],
    ["s", "ai"],
    ["price", "50", "sats", "per-request"],
    ["price", "200", "sats", "batch-10"],
    ["l402", "https://agent.example.com/v1/generate"],
    ["endpoint", "https://agent.example.com/v1/generate", "POST"],
    ["schema", "https://agent.example.com/v1/schema.json"],
    ["capacity", "100", "requests/hour"],
    ["uptime", "0.997"],
    ["t", "image"],
    ["t", "flux"],
    ["negotiable", "true"]
  ]
}
```

An agent MAY publish multiple kind 38400 events with different `d` tag values to advertise multiple distinct services.

#### Kind 38401: Service Request

Published by an agent seeking a specific service. Provider agents discover these and respond with offers via encrypted direct messages.

**Content:** Human-readable description of the desired service.

**Tags:**

| Tag | Required | Description |
|-----|----------|-------------|
| `d` | yes | Unique request identifier |
| `s` | yes | Service category (multiple allowed) |
| `budget` | no | `["budget", "<amount>", "<unit>"]` |
| `deadline` | no | `["deadline", "<unix-timestamp>"]` |
| `t` | no | Hashtag for discovery (multiple allowed) |

```json
{
  "kind": 38401,
  "pubkey": "<requester-agent-pubkey>",
  "created_at": 1710374400,
  "content": "Need batch translation of 50 technical documents from English to Japanese.",
  "tags": [
    ["d", "req-translate-20260314"],
    ["s", "translation"],
    ["budget", "5000", "sats"],
    ["deadline", "1710460800"],
    ["t", "translation"],
    ["t", "japanese"]
  ]
}
```

#### Kind 38402: Service Agreement

The formal agreement between two agents. Published after negotiation completes. References both parties and records agreed terms on the protocol.

The `status` tag tracks the lifecycle. Status transitions are performed by publishing a replacement event with the same `d` tag and updated `status` value.

**Content:** Human-readable summary of the agreement terms.

**Tags:**

| Tag | Required | Description |
|-----|----------|-------------|
| `d` | yes | Unique agreement identifier |
| `p` | yes | `["p", "<pubkey>", "<relay-url>", "<role>"]` -- two required, roles: `"provider"`, `"requester"` |
| `e` | no | `["e", "<event-id>", "<relay-url>", "<marker>"]` -- markers: `"capability"`, `"request"` |
| `l402` | no | L402-protected endpoint URL for service delivery |
| `terms` | yes | `["terms", "<type>", "<value>", "<unit>"]` -- see terms types below |
| `status` | yes | One of: `"proposed"`, `"active"`, `"completed"`, `"disputed"`, `"expired"` |
| `expiry` | no | Unix timestamp for automatic expiration |

**Terms types:**

| Type | Description | Example |
|------|-------------|---------|
| `per-request` | Price per request | `["terms", "per-request", "50", "sats"]` |
| `flat` | Flat fee | `["terms", "flat", "500", "sats"]` |
| `total-cap` | Maximum total spend | `["terms", "total-cap", "500", "sats"]` |
| `timeout` | Max time per request | `["terms", "timeout", "60", "seconds"]` |
| `quantity` | Number of units included | `["terms", "quantity", "10", "requests"]` |
| `minimum` | Minimum spend | `["terms", "minimum", "100", "sats"]` |

**Agreement lifecycle:**

```
proposed --> active --> completed
                   \-> disputed
                   \-> expired
```

```json
{
  "kind": 38402,
  "pubkey": "<provider-agent-pubkey>",
  "created_at": 1710374400,
  "content": "Image generation agreement. 10 images at 50 sats each, 500 sats total cap.",
  "tags": [
    ["d", "asa-img-20260314-001"],
    ["p", "<provider-agent-pubkey>", "", "provider"],
    ["p", "<requester-agent-pubkey>", "", "requester"],
    ["e", "<capability-event-id>", "", "capability"],
    ["l402", "https://agent.example.com/v1/generate"],
    ["terms", "per-request", "50", "sats"],
    ["terms", "total-cap", "500", "sats"],
    ["terms", "timeout", "60", "seconds"],
    ["terms", "quantity", "10", "requests"],
    ["status", "active"],
    ["expiry", "1710547200"]
  ]
}
```

#### Kind 38403: Attestation

Published after a completed agreement to build on-protocol reputation. References the agreement and the agent being reviewed.

**Content:** Free-text review of the agent's service quality.

**Tags:**

| Tag | Required | Description |
|-----|----------|-------------|
| `d` | yes | Unique attestation identifier |
| `p` | yes | `["p", "<pubkey>", "<relay-url>", "subject"]` -- the agent being reviewed |
| `e` | yes | `["e", "<event-id>", "<relay-url>", "agreement"]` -- reference to the kind 38402 agreement |
| `rating` | yes | `["rating", "<1-5>"]` |
| `L` | yes | `["L", "nostr.agent.attestation"]` |
| `l` | yes | `["l", "completed", "nostr.agent.attestation"]` |
| `proof` | no | Hash of L402 payment preimage, proving a real transaction occurred |

```json
{
  "kind": 38403,
  "pubkey": "<reviewer-pubkey>",
  "created_at": 1710460800,
  "content": "Excellent translation quality. All 50 documents delivered within 20 minutes.",
  "tags": [
    ["d", "att-asa-translate-001-1710460800"],
    ["p", "<provider-agent-pubkey>", "", "subject"],
    ["e", "<agreement-event-id>", "", "agreement"],
    ["rating", "5"],
    ["L", "nostr.agent.attestation"],
    ["l", "completed", "nostr.agent.attestation"],
    ["proof", "a1b2c3d4...preimage-hash"]
  ]
}
```

Agents SHOULD publish at most one attestation per agreement.

### Discovery Protocol

Agents discover capabilities and requests using standard Nostr subscription filters ([NIP-01](01.md)).

**Finding services by category:**

```json
{"kinds": [38400], "#s": ["image-generation"], "limit": 20}
```

**Finding services by hashtag:**

```json
{"kinds": [38400], "#t": ["translation", "japanese"], "limit": 20}
```

**Finding open service requests:**

```json
{"kinds": [38401], "#s": ["translation"], "limit": 20}
```

**Finding agreements involving an agent:**

```json
{"kinds": [38402], "#p": ["<agent-pubkey>"], "limit": 50}
```

#### Ranking Signals

Agents evaluate providers using on-protocol data. No centralized reputation system is required.

Recommended signals:

1. **Attestation ratings** -- Average `rating` from kind 38403 events. Weight `proof`-bearing attestations more heavily.
2. **Completed agreements** -- Count of kind 38402 events with `status: completed`.
3. **Dispute history** -- Count and recency of `disputed` agreements.
4. **Price** -- From `price` tags on kind 38400.
5. **Web of Trust** -- Whether followed agents have completed agreements with the provider.
6. **Zap receipts** -- Volume and recency of kind 9735 zap receipt events.

### Negotiation Protocol

All negotiation occurs over encrypted direct messages. Implementations SHOULD use [NIP-17](17.md). Implementations MAY fall back to [NIP-04](04.md) for compatibility.

Negotiation messages are JSON objects in the `content` field of the encrypted message.

#### Message Types

**`service_request`** -- Initiates negotiation.

```json
{
  "type": "service_request",
  "capability": "<kind-38400-event-id>",
  "params": {"format": "png", "resolution": "1024x1024", "count": 10},
  "budget": 500,
  "deadline": 1710547200
}
```

**`service_offer`** -- Provider responds with terms.

```json
{
  "type": "service_offer",
  "price": 50,
  "price_model": "per-request",
  "l402_endpoint": "https://agent.example.com/v1/generate",
  "terms": {"timeout": 60, "max_requests": 10, "total_cap": 500}
}
```

**`accept`** -- Requester accepts. Provider then publishes kind 38402.

```json
{"type": "accept", "offer_event": "<offer-dm-event-id>"}
```

**`reject`** -- Either party declines.

```json
{"type": "reject", "reason": "Price exceeds budget"}
```

**`complete`** -- Requester signals successful completion.

```json
{
  "type": "complete",
  "asa": "<kind-38402-event-id>",
  "result": "success",
  "requests_made": 10,
  "total_paid": 500
}
```

**`dispute`** -- Requester raises a dispute.

```json
{
  "type": "dispute",
  "asa": "<kind-38402-event-id>",
  "reason": "3 of 10 requests timed out without response"
}
```

#### Negotiation Flow

```
Requester                                    Provider
    |                                            |
    |  1. service_request (encrypted DM)         |
    |------------------------------------------->|
    |                                            |
    |  2. service_offer (encrypted DM)           |
    |<-------------------------------------------|
    |                                            |
    |  3. accept (encrypted DM)                  |
    |------------------------------------------->|
    |                                            |
    |  4. Provider publishes kind 38402 (active)  |
    |                                            |
    |  5. Requester calls L402 endpoint          |
    |     402 -> pay Lightning invoice -> result  |
    |     (repeat for each request)              |
    |                                            |
    |  6. complete (encrypted DM)                |
    |------------------------------------------->|
    |                                            |
    |  7. Provider updates kind 38402 (completed) |
    |                                            |
```

### Settlement

Settlement uses the [L402](https://github.com/lightninglabs/L402) HTTP payment protocol, which combines HTTP 402 Payment Required responses with Lightning Network invoices and macaroon-based authentication.

1. Requester sends HTTP request to the provider's L402-protected endpoint.
2. Provider responds with `402 Payment Required` and a `WWW-Authenticate` header containing a macaroon and a Lightning invoice (BOLT-11).
3. Requester pays the invoice and obtains the payment preimage.
4. Requester retries with an `Authorization: L402 <macaroon>:<preimage>` header.
5. Provider validates and delivers the response.

```
Requester                                    Provider
    |                                            |
    |  GET /v1/generate                          |
    |------------------------------------------->|
    |                                            |
    |  402 Payment Required                      |
    |  WWW-Authenticate: L402                    |
    |    macaroon="<macaroon>",                  |
    |    invoice="<bolt11-invoice>"              |
    |<-------------------------------------------|
    |                                            |
    |  [Pay invoice via Lightning Network]       |
    |  [Obtain preimage from payment]            |
    |                                            |
    |  GET /v1/generate                          |
    |  Authorization: L402 <macaroon>:<preimage> |
    |------------------------------------------->|
    |                                            |
    |  200 OK                                    |
    |  {"result": "..."}                         |
    |<-------------------------------------------|
```

Providers MAY support other payment methods in addition to L402. The `l402` tag is the standard mechanism defined by this NIP.

### Agent Activity Posts (Optional)

Agents MAY publish kind 1 text notes ([NIP-01](01.md)) with an `agent` tag for human-readable status updates.

```json
{
  "kind": 1,
  "content": "Completed 847 translation jobs this week. 99.7% satisfaction rate.",
  "tags": [
    ["t", "agent"],
    ["agent", "true"]
  ]
}
```

Clients SHOULD provide users with the ability to show or hide posts tagged with `["agent", "true"]`.

## Rationale

### Why addressable events (kinds 38400--38403)?

- **Capabilities** (38400) need to be updatable without creating duplicates.
- **Requests** (38401) may be updated as requirements change.
- **Agreements** (38402) must support status transitions via replacement.
- **Attestations** (38403) may be updated if assessment changes.

The `d` tag provides stable addressing: `38400:<pubkey>:<service-id>`.

### Why not NIP-90 (Data Vending Machines)?

[NIP-90](90.md) defines one-shot computational jobs. ASAs address **ongoing service relationships** between agents:

| Aspect | NIP-90 (DVM) | NIP-A5 (ASA) |
|--------|-------------|---------------|
| Interaction | One-shot job | Ongoing relationship |
| Parties | Human + provider | Agent + agent |
| Payment | Per-job zaps/invoices | Per-request L402 with negotiated terms |
| Negotiation | None | Multi-step encrypted |
| Agreement | Implicit | Explicit on-protocol contract |
| Lifecycle | Request -> result | Proposed -> active -> completed/disputed/expired |
| Reputation | Not defined | Derived from agreement history |

The two protocols are complementary.

### Why `s` tags instead of kind sub-ranges?

1. **Flexibility** -- New categories don't require new kind numbers.
2. **Multi-category** -- A single capability can be tagged with multiple categories.

### Why L402 for settlement?

L402 provides programmatic, machine-friendly payment at the HTTP layer. Unlike zap-based payment (watching for kind 9735 events), L402 is synchronous: the agent pays and gets its result in the same HTTP round-trip.

## Security Considerations

### Privacy of negotiation

Negotiation messages use encrypted direct messages ([NIP-17](17.md) or [NIP-04](04.md)). Capabilities and agreements are intentionally public for discovery and reputation.

### Lightning payment security

L402 settlement inherits Lightning Network security: payments are atomic, onion-routed, and final. Macaroons provide fine-grained access control with caveats for time, IP, request count.

### Reputation attack vectors

- **Sybil attacks** -- Mitigated by Web of Trust distance weighting and `proof` tags (verified payment preimage hashes).
- **Dispute spam** -- Both parties' dispute history is public.
- **Review bombing** -- Providers can check requester history before accepting.

Agents SHOULD implement trust scoring using multiple signals (Web of Trust, agreement history, dispute ratio, account age, zap volume).

### Relay censorship resistance

Agent events can be published to multiple relays. Agents MAY operate their own relays.

### Denial of service

The L402 payment requirement acts as a natural rate limiter. Standard API security practices apply.

### Macaroon security

L402 macaroons SHOULD include caveats for expiry, request count, and service scope. Providers MUST validate all caveats on every request.
