NIP-A1
======

Testimonials
------------

`draft` `optional`

## Overview

Testimonials are public endorsements written by one user about another, shown on the recipient's profile:

1. **Sender** signs a `kind:63` testimonial and delivers it privately to the recipient via an encrypted `kind:64` envelope
2. **Recipient** reviews and accepts or rejects
3. If accepted: the `kind:63` is published and referenced from the recipient's `kind:10064` curation list
4. If rejected: nothing is published, leaving no public trace

## Testimonial Event (kind 63)

```jsonc
{
  "kind": 63,
  "pubkey": "<sender-pubkey>",
  "created_at": <timestamp>,
  "tags": [["p", "<recipient-pubkey>"]],
  "content": "You've always been such an inspiring person to me!",
  "id": "<event-id>",
  "sig": "<sender-signature>"
}
```

- `content`: free text; MUST be ≤2000 characters (anti-spam, relay-enforceable). Display length is a client concern
- `p` tag: recipient pubkey (required)
- `l` tag: MAY include a [BCP-47](https://www.rfc-editor.org/info/bcp47) language code (e.g. `["l", "pt-BR"]`) to aid filtering and display
- Event MUST be signed by the author

Testimonials are immutable once accepted. To edit: author issues a `kind:5` for the old event and sends a new one through the normal flow.

## Encrypted Envelope (kind 64)

The sender NIP-44-encrypts `JSON.stringify(kind63Event)` to the recipient and publishes:

```jsonc
{
  "kind": 64,
  "pubkey": "<sender-pubkey>",
  "created_at": <timestamp>,
  "tags": [
    ["p", "<recipient-pubkey>"],
    ["expiration", "<unix-ts>"]
  ],
  "content": "<nip44 ciphertext of signed kind:63>",
  "sig": "<sender-signature>"
}
```

Clients SHOULD:

- Include `expiration` ([NIP-40](40.md)). Lifetime SHOULD be between 7 and 90 days: long enough to reach offline devices, short enough to limit exposure from a leaked recipient key
- Publish to the recipient's [NIP-17](17.md) DM relays (`kind:10050`) when available, falling back to the recipient's [NIP-65](65.md) outbox relays (`kind:10002`)
- Keep a local copy, since envelopes are not retrievable after expiration

### Receiving

1. Decrypt `content` with `conversationKey(recipientSk, envelope.pubkey)`
2. Parse as a `kind:63` and verify its signature
3. Ensure inner `pubkey` equals envelope `pubkey` (blocks replay of someone else's testimonial)
4. Ensure inner `p` tag is the recipient
5. If the inner id is in the local blacklist (see [Rejection](#rejection)), discard silently
6. Present to the user

Clients MUST reject on any verification failure.

## Testimonials List (kind 10064)

Accepted testimonials are referenced from a replaceable `kind:10064`:

```jsonc
{
  "kind": 10064,
  "pubkey": "<user-pubkey>",
  "created_at": <timestamp>,
  "tags": [
    ["e", "<kind-63-id>", "<relay-hint>", "<author-pubkey>"]
  ],
  "content": "",
  "sig": "<signature>"
}
```

The `e` tag SHOULD include both hints:

- 3rd element (relay hint): a relay that carries the `kind:63`, so clients don't need to chase the author's outbox for every testimonial
- 4th element (author pubkey): so clients can fetch `kind:0` metadata in parallel

Clients MAY soft-cap the displayed set.

### Accepting

1. Publish the signed `kind:63` to the recipient's write relays. Clients MAY also publish to the author's write relays (when known via [NIP-65](65.md)) for redundancy
2. Append an `e` tag to `kind:10064` (order is not significant; clients sort at display time)
3. Republish the updated `kind:10064`

### Removing

- **By recipient:** remove the `e` tag and republish. Display-level only; the `kind:63` remains on relays
- **By author (published testimonial):** issue a `kind:5` for the `kind:63`
- **By author (pending envelope):** issue a `kind:5` for the `kind:64` to pull it from DM relays. Useful before acceptance; afterward the public `kind:63` is what matters

Clients SHOULD hide testimonials with a matching `kind:5` deletion.

## Rejection

Rejection is a local action. Nothing about a rejection is published.

- The client discards the envelope. The sender MAY issue a `kind:5` on the envelope at any time to remove it from DM relays (it's signed by the sender's long-term key, so NIP-09 applies)
- Clients SHOULD maintain a local blacklist of rejected `kind:63` ids to prevent re-prompts on reload

**Cross-device tradeoff:** the blacklist is local, so switching clients may re-surface already-rejected envelopes until their `expiration` fires. This matches the current UX baseline for encrypted messages on Nostr. A future revision may add an opt-in synced list.

## Profile Display

- MUST render only `kind:63` events referenced in the user's current `kind:10064`. Unlisted events bypass consent
- MUST verify `kind:63` signatures and hide events with a matching `kind:5`
- SHOULD resolve the author profile via the pubkey hint in the `e` tag
- MAY paginate or sort by date or author

## Spam Mitigation

Clients MAY require proof-of-work ([NIP-13](13.md)), ignore senders in the recipient's mute list ([NIP-51](51.md)), or rate-limit by sender pubkey at the relay.

## Privacy and Security

- **Attributable by design.** Testimonials are signed with the author's long-term key, making them cryptographic proof of authorship. This is intentional: a testimonial only has value if its origin is verifiable. Authors should treat writing one as a public, non-repudiable statement
- **Full sender control over the envelope.** Because `kind:64` is signed by the sender's long-term key, the sender can delete it from DM relays at any time via standard [NIP-09](09.md) `kind:5`, before or after the recipient sees it
- **Authors can retract the published testimonial** via `kind:5` on the `kind:63`
- **Encryption protects content and rejections** (via non-publication). It does not hide the `sender → recipient` relation from the DM relay. This is acceptable here, since acceptance makes that relation public anyway
- **No forward secrecy.** A leaked recipient key exposes all retained envelopes. Short expirations mitigate but don't eliminate this

## Limitations

- **Retraction is best-effort.** A `kind:5` on a `kind:63` removes it from compliant relays, but clients that already cached the event locally will continue to render it until they re-check. Clients SHOULD periodically re-query `kind:5` deletions for the authors they display (e.g. on profile reload)
- **No right-to-be-forgotten.** Out-of-band copies (screenshots, third-party caches, archival services) are unreachable by protocol. Authors MUST treat publication as permanent in practice
- **No transport guarantee for envelopes.** A `kind:64` may expire before the recipient's device comes online. Senders SHOULD keep a local copy and be prepared to re-send
- **Sybil resistance is out of scope.** Nothing at the protocol level prevents an attacker from creating many pubkeys and flooding a recipient. Mitigation is policy; see [Spam Mitigation](#spam-mitigation)

## Example

Alice signs a `kind:63` endorsing Bob, NIP-44-encrypts it to him, and publishes a `kind:64` envelope to his DM relays.

Bob's client decrypts the envelope, verifies Alice's signature, and shows the preview. Bob clicks Accept: the client publishes the `kind:63` to his write relays and appends an `e` tag to his `kind:10064`.

Carol visits Bob's profile. Her client fetches his `kind:10064`, resolves each `e` tag to a `kind:63`, verifies signatures, and renders:

> _"Bob helped me understand Nostr when I was just getting started. Forever grateful!"_
> — Alice

If Bob had rejected, the client would have discarded the envelope and added its id to a local blacklist. Nothing published.
