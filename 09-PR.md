# NIP-09-PR

## Deletion Request by Gift Wrap Recipient

`draft` `optional`

This NIP extends [NIP-09](09.md) to allow the recipient of a gift wrap event (`kind:1059`, as defined in [NIP-59](59.md) and used by [NIP-17](17.md)) to request its deletion.

## Motivation

NIP-09 allows an event's author to request deletion by publishing a `kind:5` event whose `pubkey` matches the `pubkey` of the event to be deleted. This works for events that a user creates and controls.

Gift wraps (`kind:1059`) are different: each recipient receives their own separately-wrapped copy, but the `pubkey` on the gift wrap is an ephemeral random key, not the recipient's. This makes it impossible for recipients to request deletion of gift wraps addressed to them under standard NIP-09.

## Extension

A `kind:5` deletion request MAY reference a `kind:1059` event even when the `pubkey` of the deletion request does not match the `pubkey` of the gift wrap, provided that:

1. The `pubkey` of the deletion request appears in a `p` tag of the referenced `kind:1059` event.
2. The `k` tag of the deletion request is `"1059"`.

In all other respects the deletion request follows NIP-09 exactly.

Example:

```jsonc
{
  "kind": 5,
  "pubkey": "<recipient-pubkey>",
  "tags": [
    ["e", "<gift-wrap-event-id>"],
    ["k", "1059"]
  ],
  "content": "removing my copy of this conversation",
  // other fields...
}
```

## Semantics

The deletion request signals that the recipient no longer wishes the gift wrap to be stored or served to them. It does not affect gift wraps addressed to other participants; each recipient is responsible only for their own copy.

## Client Usage

Clients SHOULD honor a `kind:5` deletion request against a `kind:1059` event when the `pubkey` of the deletion request appears in a `p` tag of the referenced gift wrap. Clients SHOULD treat the referenced gift wrap as deleted for that user.

A client MUST NOT honor such a deletion request if the `pubkey` does not appear in a `p` tag of the referenced gift wrap.

Clients SHOULD broadcast the deletion request to all relays that hold the referenced gift wrap.

## Relay Usage

Relays SHOULD delete or stop serving a `kind:1059` event when a `kind:5` deletion request is received whose `pubkey` appears in a `p` tag of that gift wrap.

Relays MAY enforce this check before acting on the deletion request, but are not required to since they may not have knowledge of all referenced events.

Relays SHOULD continue to serve the deletion request event itself, consistent with standard NIP-09 relay behavior.

## Interaction with NIP-09

This extension applies only to `kind:1059` gift wrap events. For all other event kinds, the standard NIP-09 rule holds: only the event's author (matching `pubkey`) may request deletion.

This extension does not change the semantics of a deletion request against a gift wrap initiated by the gift wrap's own `pubkey` (the ephemeral author), which remains valid under standard NIP-09.
